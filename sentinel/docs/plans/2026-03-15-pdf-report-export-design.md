# PDF Binary Report Export — Design Document

## Goal

Add a production-grade PDF rendering pipeline so that the 6 existing React report templates (compliance summary, audit evidence, executive, NIST profile, HIPAA assessment, IP attribution) can be exported as downloadable PDFs with support for batch export, email delivery, and real-time render status.

## Architecture Overview

**Rendering engine:** `@react-pdf/renderer` v4.3 (already installed in `packages/compliance`). Server-side `renderToBuffer()` produces PDF bytes in the existing report-worker process — no headless browser needed.

**Async pipeline:**
1. Client POSTs to `/v1/reports` → API validates, creates DB row (`status: queued`), publishes to Redis stream `sentinel.reports`
2. Report-worker consumes message → gathers data, renders PDF via React-PDF, uploads to S3
3. Client polls status via SSE at `/v1/reports/:id/status` (backed by Redis pub/sub)
4. Download via `GET /v1/reports/:id/download` → 302 redirect to pre-signed S3 URL (15-min expiry)

**No binary proxying** — the API never streams PDF bytes through itself; it always redirects to S3.

## Report Registry & Template System

### ReportTemplate Interface

```typescript
interface ReportTemplate<TData = unknown> {
  type: string;                                    // e.g. "compliance_summary"
  displayName: string;                             // e.g. "Compliance Summary Report"
  description: string;
  gather(ctx: GatherContext): Promise<TData>;      // fetch data from DB
  render(data: TData, branding: BrandingContext): ReactElement; // return React-PDF document
}
```

### Registry

Self-registering pattern — each template file calls `registerTemplate(template)` at module scope. The registry maps `type → ReportTemplate` and provides `getTemplate(type)`, `listTemplates()`.

## Data Pipeline & Storage

### Hybrid Data Strategy

**Eager materialization** (fetched upfront in a Serializable transaction):
- Scan metadata, risk scores, verdict, certificate info
- Control mappings and compliance percentages
- Summary statistics and aggregations

**Paginated cursors** (streamed in chunks during render):
- Evidence records (can be thousands per scan)
- File-level attributions (IP report)
- Individual findings with details

This prevents OOM on large datasets while keeping simple reports fast.

### Storage

**Interface:**
```typescript
interface ReportStorage {
  upload(key: string, buffer: Buffer, contentType: string): Promise<void>;
  getSignedUrl(key: string, expiresInSeconds: number): Promise<string>;
  delete(key: string): Promise<void>;
}
```

**Two implementations:**
- `S3ReportStorage` — production, uses `@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner`
- `LocalReportStorage` — development, writes to `data/reports/` directory, serves via Express static

Selected by `REPORT_STORAGE=s3|local` env var.

**S3 key format:** `reports/{orgId}/{reportId}.pdf`

**Lifecycle:** S3 lifecycle policy expires objects after 90 days. DB column `expiresAt` tracks this for UI display.

### Database Changes

New columns on the existing `Report` model:
- `fileSize` (Int, nullable) — PDF size in bytes
- `pageCount` (Int, nullable) — number of pages
- `expiresAt` (DateTime, nullable) — when the S3 object expires
- `storageKey` (String, nullable) — S3 object key
- `batchId` (String, nullable) — groups batch exports
- `error` (String, nullable) — error message on failure

## API Endpoints & Delivery

### Endpoints

**POST `/v1/reports`** — Request report generation
```json
{
  "type": "compliance_summary",
  "params": { "scanId": "scan-123", "projectId": "proj-456" },
  "batchId": "batch-789"  // optional, for batch exports
}
```
Returns `201 { id, status: "queued" }`.

**GET `/v1/reports/:id/status`** — SSE stream
Events: `queued`, `rendering` (with progress %), `uploading`, `completed` (with downloadUrl), `failed` (with error).
Backed by Redis pub/sub channel `report:{id}:status`.

**GET `/v1/reports/:id/download`** — Download
Returns `302` redirect to pre-signed S3 URL (15-min expiry). Returns `404` if not ready, `410` if expired.

**GET `/v1/reports`** — List reports (paginated, filtered by org)

### Batch Export

Client sends multiple POST requests with the same `batchId`. Each is processed independently by the worker. The UI tracks batch progress by querying reports with that `batchId`.

Batch status is derived (not stored): `completed` if all done, `partial` if some failed, `failed` if all failed, `in_progress` otherwise.

### Email Delivery

After render completes, if the report request included `"delivery": "email"`:
- Worker publishes to `sentinel.notifications` stream with report metadata
- Notification-worker picks it up:
  - If PDF < 10MB: attach directly to email
  - If PDF >= 10MB: include download link (pre-signed URL with 7-day expiry)

## Error Handling & Resilience

### Render Failures

- Worker wraps `renderToBuffer()` in try/catch — on failure, updates report row with `status: 'failed'` and `error` message, publishes SSE `render.failed` event
- Retry: 3 attempts with exponential backoff (1s, 4s, 16s) via existing `withRetry` from `@sentinel/events`
- After max retries: mark as `permanently_failed`, notify via `sentinel.notifications`

### Memory & Timeout Protection

- `--max-old-space-size=512` on report-worker process
- Render timeout: 60s default, configurable via `RENDER_TIMEOUT_MS` env var
- Batch reports processed sequentially within a worker (no parallel renders) to avoid memory spikes

### S3 Upload Failures

- 3x retry with backoff before marking failed
- Orphaned S3 objects (upload succeeded, DB update failed) cleaned by daily sweep — objects older than 24h with no matching DB row

### Batch Partial Failures

- Each report in a batch is independent — individual failures don't block others
- SSE publishes per-report status within the batch

### Data Consistency

- Report row created with `status: 'queued'` *before* publishing to Redis stream
- If stream publish fails, row stays queued; a sweep marks stale `queued` reports (>5min old) as failed
- Idempotency: consumer checks if report already `completed` before processing (handles redelivery)

## Testing & Rollout

### Testing Strategy

**Unit tests:**
- Report registry: registration, lookup, validation
- Data gatherers: mock DB, verify correct data shapes
- Storage providers: mock S3 client, verify upload/download/pre-signed URLs
- Branding context: verify defaults when org has no customization

**Integration tests:**
- End-to-end render: `renderToBuffer()` with fixture data for all 6 templates — assert valid PDF (magic bytes `%PDF-1.`, page count > 0, size within range)
- Worker flow: publish to stream, assert row transitions `queued → rendering → uploading → completed`
- Batch export: request 3 reports, verify independent completion
- SSE status: connect to endpoint, trigger render, verify event sequence

**Visual regression:**
- Generate sample PDFs with known fixtures as reference — manual comparison initially
- Automated pixel-diff deferred to later iteration

### Rollout Phases

1. **Phase 1:** Single report type (Compliance Summary) behind feature flag `PDF_EXPORT_ENABLED=false`. Deploy to staging, validate end-to-end.
2. **Phase 2:** Enable all 6 report types. Add batch export.
3. **Phase 3:** Email delivery integration. Scheduled report generation (cron-triggered).

### Feature Flag

Environment variable `PDF_EXPORT_ENABLED` on API — when disabled, `POST /v1/reports` returns 404. Report worker runs but skips processing (graceful no-op).
